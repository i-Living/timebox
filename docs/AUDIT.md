## 1. Обзор проекта

TimeBox — PWA для организаторов групповых занятий: создание временных слотов, бронирование, учёт посещаемости, интеграция с Google Calendar.

**Архитектура:**
- SPA без роутера — переключение табов через состояние `tab`
- Все данные в localStorage — CRUD через `store.ts`
- Google Calendar OAuth через GIS token client (client-side только)
- Деплой в GitHub Pages через GitHub Actions

**Состав:**
- 9 компонентов (`src/components/`)
- 4 утилиты (`src/utils/`)
- 1 файл типов
- 1 файл стора
- Wiki-документация на 17 страниц

## 2. Найденные проблемы

---

### H-01: ICS-файлы генерируются с неверными разделителями строк ~~🔴 Fixed~~ ✅

- **Категория:** Правильность
- **Критичность:** ~~High~~ Fixed
- **Локация:** `src/utils/ics.ts:56, 96`
- **Проблема:** В закоммиченной версии было `'\\r\\n'` (строка из 4 символов: `\`, `r`, `\`, `n`). ICS-формат (RFC 5545) требует реальные CRLF-байты (0D 0A).
- **Риск:** Сгенерированные .ics файлы не открываются или некорректно импортируются.
- **Исправлено:** Заменено на `'\r\n'` — в JavaScript это даёт настоящие CRLF.
- **Дополнительно:** Весь модуль `ics.ts` — мёртвый код: ни один компонент его не импортирует (см. M-04).

---

### H-02: DiaryView — неверная сортировка futureSlots ✅ Fixed

- **Категория:** Правильность
- **Критичность:** High → Fixed
- **Локация:** `src/components/DiaryView.tsx:36`
- **Проблема:** Было `a.date.localeCompare(a.date)` (сравнение с самой собой, всегда 0). Слоты сортировались только по `start`.
- **Исправлено:** `a.date.localeCompare(b.date)`. Теперь сортировка сначала по дате, потом по времени.

---

### H-03: DiaryView — неверная сортировка pastSlots (второстепенная)

- **Категория:** Правильность
- **Критичность:** High
- **Локация:** `src/components/DiaryView.tsx:31`
- **Проблема:** Аналогичная ошибка — `b.start` сравнивается с самим собой:
  ```typescript
  .sort((a, b) => b.date.localeCompare(a.date) || b.start.localeCompare(b.start));
  //                                                   ↑ сравнивается b с b, всегда 0
  ```
  Внутри одной даты порядок по времени начала — неопределённый.
- **Риск:** Слоты за один день могут отображаться в случайном порядке.

```typescript
// Исправление:
.sort((a, b) => b.date.localeCompare(a.date) || a.start.localeCompare(b.start));
```

### M-02: Date через UTC → локальные компоненты ✅ Fixed

- **Категория:** Правильность
- **Критичность:** Medium → Fixed
- **Проблема:** `new Date('2026-06-20')` интерпретируется как UTC-полночь. Затем `toISOString().slice(0,10)` работает в UTC. Для пользователей с отрицательным смещением (UTC−5 и западнее) дата может сместиться на предыдущий день.
- **Риск:** Для российских пользователей (UTC+3) ошибки нет. Но если приложение расширяется на другие регионы — даты поедут.
- **Рекомендация:** Заменить на локальные Date-методы:

```typescript
export function today(): string {
  const d = new Date();
  return d.getFullYear() + '-' +
    String(d.getMonth() + 1).padStart(2, '0') + '-' +
    String(d.getDate()).padStart(2, '0');
}
```

### M-03: Валидация capacity ✅ Fixed

- **Категория:** Безопасность
- **Критичность:** Medium → Fixed
- **Проблема:** Поле `capacity` ограничено `min=1 max=50` только через HTML-атрибуты. Через DevTools можно сохранить слот с capacity = 9999.
- **Риск:** Переполнение localStorage большими данными, некорректная работа `getFreeSlots` / `getBookedCount`.
- **Рекомендация:** Добавить ограничение при сохранении:

```typescript
const handleSave = () => {
  const safeCapacity = Math.min(50, Math.max(1, capacity));
  onSave({
    // ...
    capacity: safeCapacity,
    // ...
  });
};
```

### L-01: strict: true в tsconfig ✅ Fixed

- **Категория:** Качество
- **Критичность:** Low → Fixed
- **Проблема:** Отсутствует `"strict": true`. TypeScript 6 по умолчанию включает `strictNullChecks` при `strict: true`. Без этого `null` и `undefined` могут быть неявно присвоены.
- **Риск:** Потенциальные `undefined is not an object` ошибки в рантайме.

---

### L-02: XSS-поверхность для GCal токена

- **Категория:** Безопасность
- **Критичность:** Low
- **Локация:** `src/utils/gcal.ts:110`
- **Проблема:** OAuth-токен Google хранится в localStorage. При XSS-атаке токен может быть украден. Preact JSX экранирует значения, но сторонние скрипты или расширения могут прочитать localStorage.
- **Риск:** Стандартный риск для client-side OAuth в SPA. Принят.

---

### L-03: Мемоизация expandSlots ✅ Fixed

- **Категория:** Производительность
- **Критичность:** Low → Fixed
- **Проблема:** `expandSlots` вызывается дважды при каждом рендере (для недели и для дня). При большом количестве слотов (100+) может вызывать задержки.
- **Рекомендация:** Мемоизировать:

```typescript
const weekSlots = useMemo(() => expandSlots(data.slots, weekStart, weekEnd), [data.slots, weekStart]);
```

(Требует импорта `useMemo` из preact/hooks.)

## 4. Что не вошло в аудит (субъективные замечания)

- **Нет тестов** — для личного проекта ок, но при росте кода стоит добавить vitest.
- **Нет миграции данных** — `OrganizerData.version = 1` декларируется, но не проверяется при загрузке. Если формат изменится, старые данные могут сломаться без сообщения пользователю.
- **Нет подтверждения выхода** — если пользователь нажмёт «Назад» или закроет вкладку с несохранёнными изменениями, данные не теряются (всё в localStorage), но UI не предупреждает.
- **Нет лимита на длину имени клиента/контакта** — переполнение localStorage через длинные строки технически возможно.
- **Share Target API** — обрабатывает `text` и `url` из URL-параметров, но `title` игнорируется (есть в манифесте PWA, но не в коде).
